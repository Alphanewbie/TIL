# AWS

- Amazon Route 53
    - AWS에서 제공하는 DNS 웹 서비스. DNS란 도메인 네임(www.example.com)을 컴퓨터가 소통하는데 사용하는 IP 주소로 변환하여 주는 역할을 말한다.
    - 다음과 같은 주요 기능
        - 도메인 기능, DNS관리, 헬스 체크, 트래픽 관리, 사설 DNS(VPC내의 사설 리소스에 대한 DNS관리)
- VPC
    - AWS에서 제공하는 네트워킹 서비스, AWS 클라우드 내에 가상 네트워크 환경을 생성할 수 있다. VPC안에 EC2, RDS, 로드 밸런서 같은 리소르를 같이 묶어서 관리할 ㅜㅅ 있다.
    - VPC는 IP 주소 범위, 서브넷, 라우팅 테이블 및 네트워크 게이트 웨이 선택을 포함하여 네트워크 아키텍쳐를 설계할 수 있는 유연성 제공, 및 인바운드 아웃바운드 트래픽 규칙을 정의하기 위해 보안 그룹 및 네트워크 액세스 제어 목록을 생성하여 리소스에 대한 액세스를 제어할 수 있다.
- AWS Transit Gateway
    - AWS에서 제공하는 네트워킹 서비스, 여러 VPC와 온 프레미스 서버를 연결하는 프로세스를 단순화한다. 즉, 네트워크 연결을 통합하고 대량의 VPC와 온 프레미스 환경간 트래픽윽 효율적으로 관리하는 허브 역할
    - 주요 이점
        - 단순화된 네트워크 아키텍쳐(중앙화된 네트워크 연결), 향산된 확장성, 강화된 보안 컴플라이언스, 성능 및 대역폭 관리 개선, 다른 AWS서비스와의 통합, 중앙 집중식 모니터링 문제 해결
- AWS Direct Connect
    - AWS에서 제공하는 네트워킹 서비스, 온프레미스 인프라와 AWS 클라우드 간 전용 사설 네트워크 연결을 설정할 수 있다. 공용 인테넛을 우회하여 AWS에 더 안정적이고 빠르게 액세스 할 수 있다.
- Direct Connect 게이트웨이 프로비저닝
    - 사내에서 AWS로의 전용 네트워크 연결을 쉽게 설정할 수 있는 클라우드 서비스 솔루션인 Amazon Direct Connect를 구성하는 단계입니다. Direct Connect 게이트웨이는 AWS 전송 게이트웨이 및 VGW(Virtual Private Gateway)와의 연결을 관리하는 데 사용할 수 있는 글로벌 리소스입니다.
- Amazon DynamoDB
    - AWS에서 관리되는 NoSQL 데이터베이스 서비스. 빠르고 예측 가능한 성능 및 원활한 확장성을 제공하도록 설계 되었고. 게임, 광고 기술, IoT, 모바일 등 저지연 같은 고처리량 데이터 액세스가 필요한 다양한 사용 사례에 적합한 애플리케이션에 적합.
- OU
    - AWS에서 OU는 조직 내의 "조직 단위"를 의미합니다. AWS 조직은 여러 AWS 계정을 중앙에서 관리하고 통합할 수 있는 서비스로써. OU(조직 단위)는 조직 내에서 AWS 계정을 그룹화하는 데 사용되므로 계정 전체에서 정책을 보다 쉽게 적용하고 관리할 수 있다.
- SCP
    - AWS에서는 서비스 제어 정책(Service Control Policy)을 의미합니다. 서비스 제어 정책은 AWS 조직 내에서 여러 AWS 계정에 걸쳐 AWS 서비스 및 리소스에 대한 액세스를 관리하고 제어하는 데 사용되는 정책 유형.
    - 조직의 관리 계정 관리자가 회원 계정에 대한 세분화된 권한과 제한을 정의할 수 있습니다. 이를 통해 특정 작업이 허용되거나 거부되도록 하여 전체 조직에서 일관된 보안 자세를 강제하고 준수 요건을 충족할 수 있습니다.
    - IAM vs SCP
        - IAM
            - AWS 서비스 및 리소스에 대한 안전한 액세스 관리를 가능하게 합니다. IAM을 사용하여 AWS 사용자와 그룹을 생성하고 관리하며, EC2 인스턴스, S3 버킷 등의 AWS 리소스에 대한 액세스를 허용하거나 거부하는 권한을 설정할 수 있습니다.
            IAM은 개별 AWS 계정 내에서 사용자 또는 역할 수준에서 권한을 허용하거나 거부하는 데 주로 사용됩니다. 이러한 권한은 어떤 리소스에 대해 어떤 작업이 허용되거나 거부되는지를 결정합니다.
        - SCP
            - AWS Organizations에서 조직 내의 모든 AWS 계정에 대한 세부적인 권한과 제어를 설정하는 데 사용됩니다.
            SCP는 권한을 부여하지 않고, 조직, 조직 단위 (OU) 또는 계정에 대한 최대 권한을 정의합니다. 다시 말해, SCP는 해당 계정의 사용자와 역할이 수행할 수 없는 작업을 제한합니다. 만약 SCP에서 특정 작업이 허용되지 않는다면, 해당 사용자는 IAM 권한이 허용해도 해당 작업을 수행할 수 없습니다.
- AWS Config
    - AWS 리소스 구성을 평가, 감사 및 평가할 수 있게 해줍니다. 리소스 구성 이력을 지속적이고 상세하게 유지하고, 리소스 변경을 모니터링하며, 원하는 구성과 전반적인 준수 여부를 확인할 수 있습니다.
- Amazon CloudFront
    - AWS에서 제공하는 콘텐츠 전송 네트워크(CDN) 서비스. network of edge locations를 통해 콘텐츠를 배포하여 최종 사용자의 대기 시간을 줄이고 로드 시간을 단축합니다.
- Fargate
    - 컨테이너용 서버리스 컴퓨팅 엔진. 가상 시스템, 클러스터 또는 확장과 같은 기본 인프라를 관리할 필요 없이 컨테이너형 애플리케이션을 배포하고 실행할 수 있다.
    - 즉, 컨테이너용 서버리스 컴퓨팅 엔진으로, 기본 인프라를 관리하지 않고도 컨테이너형 애플리케이션을 배포하고 실행할 수 있습니다.
- Amazon API Gateway
    - 애플리케이션에 대한 API를 생성, 게시, 유지, 모니터링 및 보안할 수 있는 제공되는 서비스입니다. API를 사용하면 서로 다른 소프트웨어 애플리케이션이 서로 데이터를 통신하고 공유할 수 있으며, API Gateway를 사용하면 API에 대한 액세스를 규모에 맞게 쉽게 관리하고 제어할 수 있습니다.
- ECS vs. EKS
    - 공통점
        - AWS에서 제공하는 컨테이너 오케스트레이션 서비스
    - 차이
        - ECS는 AWS 전용 플랫폼을 기반으로 구축되어 AWS 서비스와의 네이티브 통합을 제공하는 반면, EKS는 오픈 소스 쿠버네티스 플랫폼을 기반으로 하여 더 큰 휴대성과 더 넓은 생태계를 제공합니다.
- AWS Elastic Beanstalk
    - 아마존웹서비스(AWS)가 제공하는 완전 관리형 서비스로 자바를 비롯한 다양한 언어로 애플리케이션을 손쉽게 배포, 관리, 확장할 수 있도록 지원한다. 애플리케이션을 실행하는 데 필요한 인프라를 설정하고 관리하는 프로세스를 간소화하여, 배포에는 신경 쓰지 않고 개발에만 집중할 수 있도록 지원한다.
- Amazon Simple Notification Service
    - AWS에서 제공하는 메세징 서비스, SNS를 사용하면 pub-sub 패턴을 사용하여 사람, 장치 또는 기타 AWS 서비스 등의 여러 가입자에게 메시지 또는 알림을 보낼 수 있다. 모바일 푸시 알림, SMS, 이메일을 비롯한 다양한 유형의 메시지는 물론 AWS Lambda, Amazon Simple Queue Service(SQS) 또는 어플리케이션 endpoints도 지원한다.
- Redis(Remote Dictionary Server)
    - 캐시, 데이터베이스 또는 메시지 브로커로 사용할 수 있는 오픈 소스, 메모리 내 데이터 구조 저장소입니다. 고성능, 짧은 대기 시간 및 대용량 데이터를 쉽게 처리할 수 있는 기능으로 알려져 있습니다. Redis는 문자열, 해시, 목록, 집합, 정렬된 집합, 비트맵, 하이퍼로그와 같은 다양한 유형의 데이터 구조를 지원하므로 다양한 용도에 적합합니다.
- Availability Zones
    - 영역 내에서 격리된 위치로, 인프라 및 애플리케이션에 이중화, 내결함성 및 고가용성을 제공합니다. 각 AWS 영역은 여러 가용성 영역으로 구성되어 있습니다. 이 영역은 지리적으로 가깝지만 별도의 전력망에 위치한 별도의 데이터 센터이며 정전, 네트워크 연결 또는 자연 재해와 같은 문제로 인한 장애 위험을 최소화하도록 설계되었습니다.
    - Multi-AZ deployment
        - AWS에서 인프라 및 애플리케이션의 가용성, 내결함성 및 성능을 향상시키기 위해 사용되는 전략입니다. Multi-AZ 배포 환경에서는 인스턴스, 데이터베이스 또는 스토리지와 같은 리소스가 동일한 AWS 영역 내의 여러 가용성 영역에 분산됩니다. 이 설정을 통해 가용성 영역 중 하나에서 장애 또는 운영 중단이 발생한 경우에도 애플리케이션을 계속 운영할 수 있습니다.
    - Multi-AZ replication
        - AWS에서 사용되는 기술로, 단일 영역 내의 여러 AZ에 걸쳐 데이터를 자동으로 복제합니다. 이 접근 방식은 데이터의 복사본을 지리적으로 격리된 서로 다른 데이터 센터에 저장함으로써 데이터의 가용성, 내결함성 및 내구성을 향상시킵니다. 한 가용성 영역에서 장애 또는 운영 중단이 발생할 경우 다른 영역에서도 데이터에 액세스할 수 있으므로 다운타임을 최소화하고 데이터 손실 위험을 줄일 수 있습니다.
- Amazon RDS DB
    - AWS가 제공하는 관리형 데이터베이스 서비스로 클라우드에서 관계형 데이터베이스를 설정, 운영, 확장하는 과정을 간소화합니다 MySQL, Postgre, SQL, MariaDB, Oracle 및 Microsoft SQL Server를 비롯하여 Amazon의 자체 데이터베이스 엔진인 Amazon Aurora도 사용할 수 있다.
- AWS Systems Manager
    - AWS에서 제공하는 관리 서비스로, AWS 리소스 및 사내 인프라를 모니터링하고 제어할 수 있도록 지원합니다. 통합 사용자 인터페이스를 제공하여 운영 작업을 자동화하고, 일관된 구성을 적용하며, 환경 전반에 걸쳐 operational insights을 얻을 수 있습니다.
- Amazon EventBridge
    - AWS의 제공하는 서버리스 이벤트 버스 서비스로, 이벤트를 사용하여 다양한 AWS 서비스와 타사 애플리케이션을 연결하여 이벤트 기반 애플리케이션을 구축할 수 있습니다. 서로 다른 서비스 간에 이벤트를 수집, 처리 및 라우팅하는 프로세스를 간소화하여 분리되고 확장 가능하며 유지 관리 가능한 아키텍처를 만들 수 있습니다.
- Amazon EFS : AWS에서 제공하는 완벽하게 관리되고 확장 가능하며 탄력적인 네트워크 파일 시스템입니다. EFS는 Amazon EC2, AWS 람다 및 Amazon Elastic Container Service(ECS) 또는 Amazon Elastic Kubernetes Service(EKS)에서 실행되는 컨테이너와 같은 AWS 컴퓨팅 서비스와 함께 사용하도록 설계되었습니다. 여러 인스턴스와 서비스가 동시에 동일한 파일 시스템에 액세스하고 공유할 수 있으므로 공유 스토리지 솔루션이 필요한 애플리케이션에 이상적인 선택입니다.
    - Amazon EFS 와 AWS S3의 차이
        - Amazon EFS : 공유 스토리지 사용 사례를 위해 설계된 네트워크 파일 시스템으로 파일 시스템 인터페이스를 제공하는 시스템
        - AWS S3 : 대량의 데이터를 저장하고 검색하기 위해 설계된 객체 스토리지 서비스이며 데이터 액세스를 위해 RESTful API를 사용합니다.
- Lambda@Edge
    - Lambda@Edge는 CloudFront Edge 위치에서 AWS Lambda 함수를 실행하여 CloudFront가 제공하는 콘텐츠를 사용자 지정할 수 있는 Amazon CloudFront의 기능입니다. 즉, 서버를 관리할 필요 없이 CloudFront 이벤트에 따라 기능을 실행할 수 있습니다.
    Lambda@Edge 함수는 CloudFront 이벤트에 대한 응답으로 실행되므로 이 함수를 사용하여 CloudFront 동작 방식을 변경할 수 있습니다. 예를 들어, 머리글, 메서드 또는 기타 요청 속성에 따라 뷰어에 대한 CloudFront 응답을 변경할 수 있습니다.
    - 가장큰 이점으로 Lambda를 사용자에게 가깝게 실행하게 할 수 있다는 것입니다.
- OpenSearch Service
- AWS Application Discovery Agent
    - 애플리케이션 검색 서비스는 기업이 사내 데이터 센터에 대한 정보를 수집하여 애플리케이션 마이그레이션 프로젝트를 계획할 수 있도록 지원합니다. 워크로드를 더 잘 이해할 수 있도록 서버에서 구성, 사용 및 동작 데이터를 수집하고 표시합니다.
- AWS Resource Access Manager
    - AWS 계정이나 AWS 조직 내에서 AWS 리소스를 쉽고 안전하게 공유할 수 있도록 해주는 서비스입니다.
- AWS Compute Optimizer
    - AWS 리소스의 구성 및 활용률 메트릭을 분석한 다음 이러한 리소스를 최적화하여 성능과 비용 효율성을 개선하는 방법에 대한 권장 사항을 제공하는 서비스입니다.
    Compute Optimizer는 워크로드의 구성 및 리소스 활용률을 평가한 다음 활용률 패턴에 따라 최적의 AWS 리소스(EC2 인스턴스 유형 등)를 권장합니다. 기계 학습을 사용하여 과거 활용 데이터를 분석하고 더 낮은 비용으로 최적의 성능을 제공할 수 있는 이상적인 구성을 식별한다.

## 기타

- latency : 네트워크를 통해 데이터 패킷이 출발지에서 목적지까지 이동하는 데 걸리는 시간을 의미. 일반적으로 밀리초(ms)로 측정되며 데이터 전송 중 발생하는 지연을 나타낸다.
- bandwidth : 대역폭은 주어진 시간 동안 네트워크 연결을 통해 전송할 수 있는 최대 데이터량을 의미합. 일반적으로 비트/초(bps), 킬로비트/초(Kbps), 같이 초당 전송량으로 표시된다.
- Metadata : 메타데이터는 다른 데이터에 대한 정보를 제공하는 데이터입니다. 즉, 특정 데이터의 특성, 품질 또는 컨텍스트를 설명하는 설명 속성 또는 속성 집합입니다. 메타데이터는 데이터를 보다 효율적으로 구성, 검색, 이해 및 사용하는 데 도움이 됩니다.
- CDN : Content Delivery Network는 전 세계 사용자에게 빠르고 효율적인 웹 콘텐츠 전송을 제공하기 위해 함께 작동하는 분산된 서버 시스템. CDN은 최종 사용자와 지리적으로 가까운 서버에서 컨텐츠를 캐싱하고 제공함으로써 지연 시간을 줄이고 로드 시간을 개선하며 전체 사용자 환경을 향상시키도록 설계되었습니다.
- native integration : 동일한 생태계 내에서 다양한 소프트웨어 애플리케이션, 서비스 또는 구성 요소 간의 원활하고 내장된 연결성을 말합니다. 클라우드 서비스의 맥락에서 native integration은 일반적으로 동일한 클라우드 공급자 내의 서비스가 원활하게 함께 작동하도록 설계되어 있어 추가 구성이 거의 필요하지 않거나 전혀 필요하지 않음을 의미합니다.
- ecosystem : 소프트웨어 에코시스템은 함께 작동하고 서로를 지원하도록 설계된 소프트웨어 애플리케이션, 플랫폼 및 서비스의 모음을 의미합니다. 여기에는 생태계의 생성, 유지보수 및 성장에 기여하는 개발자, 사용자 및 기타 이해관계자가 포함됩니다.
- Application Load Balancer : OSI 모델의 애플리케이션 계층(계층 7)에서 작동하는 AWS에서 제공하는 로드 밸런서 유형입니다. 들어오는 애플리케이션 트래픽을 여러 EC2(Amazon Elastic Compute Cloud) 인스턴스, 컨테이너 또는 기타 리소스에 분산하여 성능을 최적화하고 고가용성을 보장하며 내결함성을 개선하도록 설계되었습니다.
- RTO (Recovery Time Objective) : 자연 재해, 하드웨어 장애 또는 사이버 공격과 같은 운영 중단 이벤트가 발생한 후 비즈니스에 허용할 수 없는 결과를 초래하기 전에 시스템, 애플리케이션 또는 서비스가 중단되거나 사용할 수 없는 최대 시간을 나타냅니다. 즉, RTO는 비즈니스 프로세스가 중단된 후 허용할 수 없는 손실이나 손상을 방지하기 위해 복구해야 하는 목표 기간입니다.
- active-active strategy : 애플리케이션이나 서비스의 두 개 이상의 인스턴스가 동시에 실행되고 요청을 처리하는 분산 시스템에서 사용되는 고가용성 및 내결함성 접근 방식입니다. 이 전략은 장애나 중단이 발생하더라도 시스템이 계속 작동할 수 있도록 보장합니다. 나머지 활성 인스턴스는 다운타임 없이 워크로드를 계속 처리할 수 있기 때문입니다.
- CIDR : Classless Inter-Domain Routing 에서는 IP 주소 뒤에는 슬래시(/)와 네트워크 접두사 길이를 나타내는 숫자가 붙습니다. 접두사 길이는 특정 네트워크에 대해 고정된 주소의 비트 수를 나타내며, 나머지 비트는 해당 네트워크 내의 개별 호스트에 사용할 수 있습니다. 예를 들어, CIDR 표기법 192.168.1.0/24는 24비트 접두사를 가진 네트워크를 나타내며, 여기서 처음 24비트(192.168.1)는 네트워크 주소를 정의하고 나머지 8비트는 해당 네트워크 내의 개별 호스트 주소에 사용할 수 있습니다.
- SaaS : Software as a Service의 약자입니다. 사용자의 로컬 컴퓨터나 장치에 설치되지 않고 인터넷을 통해 소프트웨어가 제공되는 소프트웨어 라이센스 및 제공 모델입니다. 애플리케이션은 타사 공급업체가 호스팅하고 유지 관리하며, 타사 공급업체는 소프트웨어가 항상 최신 상태이고 안전하며 사용 가능한지 확인합니다. 사용자는 일반적으로 웹 브라우저를 통해 소프트웨어에 액세스하고 사용자 수, 사용량 또는 제공된 기능과 같은 요인에 따라 가입 모델에 따라 요금이 부과됩니다.
- event bus
    - 시스템의 구성 요소들 간의 통신을 용이하게 하기 위해 사용되는 소프트웨어 아키텍처 패턴이다. 이것은 구성 요소들이 서로에 대한 직접적인 의존성이나 지식 없이 이벤트를 보내고, 수신하고, 들을 수 있는 중앙 허브 역할을 한다.
    - 이벤트 중심 아키텍처에서 구성 요소, 서비스 또는 모듈은 이벤트를 방출하고 수신함으로써 상호 작용합니다. 이벤트 버스는 해당 이벤트가 적절한 수신자에게 전달되도록 이러한 이벤트의 배포를 관리합니다. 이를 통해 구성요소가 분리되어 독립적으로 진화할 수 있으며 모듈화, 유연성 및 확장성이 향상됩니다.
    - 장점
        - Decoupling : 구성 요소들은 서로의 존재에 대해 알 필요가 없으며, 이는 시스템을 더 모듈화하고 유지 관리하기 쉽게 만든다.
        - Scalability : 이벤트 버스는 여러 인스턴스에 분산될 수 있으므로 시스템의 확장성을 향상시키는 데 도움이 됩니다.
        - Flexibility : 시스템의 나머지 부분에 영향을 미치지 않고 새 구성 요소를 추가하거나 기존 구성 요소를 수정할 수 있습니다.
        - Asynchronous communication : 이벤트를 비동기식으로 처리하여 응답성과 리소스 활용도를 높일 수 있습니다.
- CNAME : 도메인 이름을 다른 도메인 이름에 매핑하는 DNS 레코드의 한 유형입니다. CNAME 레코드의 주요 목적은 도메인에 대한 별칭을 만들어 여러 이름으로 동일한 대상 도메인을 참조할 수 있도록 하는 것입니다. 이것은 여러 하위 도메인을 단일 도메인으로 가리키거나 복잡한 도메인 이름에 대한 사람이 읽을 수 있는 별칭을 만드는 것과 같은 다양한 시나리오에서 유용할 수 있습니다.
- A record : 도메인 이름을 IPv4 주소에 매핑하는 DNS(Domain Name System) 레코드의 한 유형입니다. A 레코드의 주요 목적은 사람이 읽을 수 있는 도메인 이름을 해당 도메인과 관련된 웹 사이트 또는 서비스를 호스팅하는 서버의 IP 주소와 연결하는 것입니다. 사용자가 도메인 이름을 입력하여 웹 사이트나 서비스에 액세스하려고 하면 DNS 확인자는 해당 A 레코드를 조회하여 연결할 IP 주소를 결정합니다.
- AAAA record :도메인 이름을 IPv6 주소에 매핑하는 DNS(Domain Name System) 레코드의 한 유형입니다. AAAA 레코드의 주요 목적은 사람이 읽을 수 있는 도메인 이름을 해당 도메인과 관련된 웹 사이트 또는 서비스를 호스팅하는 서버의 IPv6 주소와 연결하는 것입니다. IPv6는 사용 가능한 IPv4 주소가 모두 사용되어 IPv4를 대체하도록 설계된 인터넷 프로토콜의 최신 버전입니다.
- web fleet : 들어오는 웹 트래픽을 처리하고 요청을 처리하며 사용자에게 콘텐츠를 전달하기 위해 함께 작동하는 서버 그룹입니다. 웹 플릿은 여러 서버 간에 워크로드를 분산시켜 웹 응용 프로그램 또는 웹 사이트의 성능, 가용성 및 신뢰성을 향상시키는 데 사용됩니다.
웹 플릿은 일반적으로 수신 요청을 플릿의 사용 가능한 서버에 분산하는 로드 밸런싱을 통해 관리됩니다. 로드 밸런싱은 라운드 로빈, 최소 연결 또는 서버 응답 시간과 같은 다양한 알고리즘을 사용하여 수행하여 단일 서버가 트래픽으로 가득 차지 않도록 할 수 있습니다.
- EIP(Elastic IP address) : 동적 클라우드 컴퓨팅을 위해 AWS(Amazon Web Services)에서 제공하는 정적 공개 IPv4 주소입니다. AWS 계정 내에서 한 인스턴스에서 다른 인스턴스로 빠르게 재맵될 수 있기 때문에 "탄력적"입니다.
Elastic IP는 특정 인스턴스가 아닌 AWS 계정과 연결되며, 사용자가 명시적으로 릴리스하도록 선택할 때까지 계정과 연결된 상태로 유지됩니다. 그러나 기존의 정적 IP 주소와 달리 Elastic IP를 사용하면 계정의 다른 인스턴스에 주소를 빠르게 재매핑하여 인스턴스 또는 가용성 영역 장애를 마스킹할 수 있습니다.
